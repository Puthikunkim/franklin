# Slice 7 — In-app notifications (outbid / won / sold / ending-soon)

**Status:** Design approved 2026-07-06
**Predecessors:** Slices 1 (hero auction flow), 2 (listing creation), 3 (dealer dashboard), 4 (discovery), 5 (buy-now), 6 (seller unpublish), all merged to `main`.

## 1. Goal

Tell a dealer when something happened that they'd want to know about but may not be
looking at: they were **outbid**, they **won** an auction, their listing **sold**, or an
auction they're **watching is ending soon**. A 🔔 in the header with an unread badge draws
them to a `/notifications` page that lists these. Until now the platform had no way to pull a
dealer back in after they navigate away — every state change was invisible unless they
happened to be on the right page.

## 2. Scope

**In scope:**
- A thin `notifications` event-log table (`outbid` / `won` / `sold`).
- Row generation *inside* the existing writer RPCs where the moment already occurs
  (`place_bid`, `close_auction`, `buy_now_listing`) — no new triggers, no polling.
- A `mark_notifications_read` writer RPC (service_role-only) + action.
- Read helpers in `src/lib/notifications.ts` (unread count, list, derived ending-soon).
- A header 🔔 with an unread-count badge, and a `/notifications` page.
- The **ending-soon** alert computed on-read from the watchlist (no stored rows, no scheduler).

**Out of scope (later / deferred):** `passed` (unsold-closure) and settlement-status
notifications, watcher notifications beyond ending-soon, realtime badge push, email/push
delivery, per-row mark-unread, and row-level security (deferred with real auth). No
credential-dependent work — this slice needs no R2 or external creds.

## 3. Product decisions (locked during brainstorming)

| Decision | Choice |
|----------|--------|
| Event types | `outbid`, `won`, `sold` (stored) + `ending-soon` (derived). No `passed`. |
| Ending-soon mechanism | Computed on-read from the watchlist (watched + `live` + ending within the window). No scheduler/cron — consistent with the platform's no-infra, phone-first ethos. |
| Ending-soon window | 30 minutes. |
| Surface | Header 🔔 with an unread badge on every page → a dedicated server-rendered `/notifications` page. |
| Read state | Opening `/notifications` marks all unread **stored** rows read (badge's stored portion clears), but rows unread *this* visit are highlighted as new. No per-row buttons. |
| Badge count | Unread stored rows **+** currently-ending-soon watched auctions. The ending-soon portion may not clear on view (it self-resolves when the auction ends) — an alert you never see isn't an alert. |
| Outbid recipient | The **displaced** previous leader only (lead actually changed hands). Not a self-raise, not a proxy-hold (that challenger already sees "outbid" in the live bid response). |
| Won / sold recipients | `won` → the winning bidder; `sold` → the seller. |
| Buy-now buyer | No `won` row — the buyer is the actor and is already redirected to `/won/[id]`. Only the seller gets `sold`. |
| Security | `notifications` is anon-**readable** (dealer-scoped in the query, like every read in this no-RLS demo) but only the SECURITY DEFINER RPCs write rows; `mark_notifications_read` is service_role-only (revoke-before-grant), cookie identity. |

## 4. Data model

New migration `supabase/migrations/0009_notifications.sql`.

```sql
create table notifications (
  id uuid primary key default gen_random_uuid(),
  recipient_dealer_id uuid not null references dealers(id),
  type text not null check (type in ('outbid','won','sold')),
  auction_id uuid not null references auctions(id),
  created_at timestamptz not null default now(),
  read_at timestamptz            -- null = unread
);
create index on notifications (recipient_dealer_id, created_at desc);
```

A thin event log — `recipient + type + auction` is enough to render every message; the page
joins to the auction/vehicle for the label and amount at render time (single source of truth,
no denormalized text that can go stale). `type` is a checked text token rather than a new enum
so it's trivial to extend later (`passed`, etc.), matching how `settlements.status` uses text.

`test_reset()` is redefined in this migration to also `truncate notifications restart identity
cascade;` so integration tests stay isolated between cases (a notification generated by one
test never leaks into the next).

## 5. Generation (inside existing RPCs)

Additive only — no change to any RPC's return shape or existing guards. A small internal
helper keeps the inserts one line at each call site:

```sql
-- inserts one notification; called only from within the SECURITY DEFINER writers,
-- so it runs with the function-owner's privileges (owner has insert on notifications).
create or replace function _notify(p_recipient uuid, p_type text, p_auction uuid)
returns void language sql as $$
  insert into notifications (recipient_dealer_id, type, auction_id)
  values (p_recipient, p_type, p_auction);
$$;
revoke execute on function _notify(uuid, text, uuid) from public, anon, authenticated;
```

The revoke is belt-and-suspenders: `_notify` is `SECURITY INVOKER`, so a direct anon call
would already fail (anon has no `INSERT` on `notifications`), but revoking the default PUBLIC
`EXECUTE` grant makes "only the definer writers create rows" an explicit, testable invariant.
The internal calls are unaffected — `place_bid`/`close_auction`/`buy_now_listing` run as the
function owner, which always retains `EXECUTE` on its own functions.

- **`place_bid`** — in the lead-change branch only (a challenger beats the current leader's
  proxy → `v_new_winner := p_dealer_id` and `a.current_winner_dealer_id` is non-null and not
  the same dealer), insert `outbid` for the **displaced** leader (`a.current_winner_dealer_id`,
  which is the pre-bid snapshot). No row when there was no prior leader, when a dealer raises
  their own proxy, or when the leader's proxy holds.
- **`close_auction`** — on the `sold` branch, insert `won` for `a.current_winner_dealer_id`
  **and** `sold` for `a.seller_dealer_id`. Passed branch inserts nothing. `close_auction` is
  already idempotent (guarded by `status <> 'live'`), so each fires exactly once.
- **`buy_now_listing`** — on success (`'bought'`), insert `sold` for `a.seller_dealer_id`.
  The buyer is the actor (redirected to `/won/[id]`), so no `won` row for them. Idempotent via
  the status guard.

## 6. Reads + derived ending-soon

All in `src/lib/notifications.ts`, following the `(sb: SupabaseClient, dealerId: string)` DI
pattern (tests pass `admin`, pages pass `await serverClient()`):

- **`getUnreadCount(sb, dealerId): Promise<number>`** — count of unread stored rows
  (`read_at is null`) **plus** the count of watched auctions currently in the ending-soon
  window. Drives the header badge (runs on every page via the server-rendered `Header`).
- **`listNotifications(sb, dealerId)`** — stored rows for the dealer, newest first, joined to
  `auction → vehicle` (limit ~50) for rendering.
- **`getEndingSoonWatched(sb, dealerId)`** — `watchlist ⋈ auctions` where `status = 'live'`
  and `end_time` is between now and now + 30 min, joined to the vehicle. JS-computed bounds
  (`new Date().toISOString()`), mirroring `getMyBiddingAuctions`. Transient — no storage, no
  read-state; naturally disappears when the auction ends or leaves the window.

The 30-minute window is a named constant in the lib.

## 7. Service + action (mark read)

- **RPC** `mark_notifications_read(p_dealer_id uuid) returns void`
  (`plpgsql security definer set search_path = public`): `update notifications set
  read_at = now() where recipient_dealer_id = p_dealer_id and read_at is null;`. Grants follow
  the writer pattern — `revoke execute ... from public, anon, authenticated;` then
  `grant execute ... to service_role;`.
- **Service** `markNotificationsRead(dealerId)` in `src/lib/notifications.ts` (or a co-located
  `service.ts`) — `serviceClient().rpc("mark_notifications_read", { p_dealer_id })`. Server-only
  (`serviceClient` never imported into a `"use client"` module).

## 8. UI

- **Header** (`src/components/Header.tsx`, already a server component) — add a 🔔 link to
  `/notifications`, logged-in only (beside Dashboard), with an unread-count badge from
  `getUnreadCount`. The badge renders only when the count is > 0.
- **`/notifications` page** (`src/app/notifications/page.tsx`, server component; redirect
  `/login` if no dealer cookie):
  1. Reads `listNotifications` (capturing which rows are unread this visit) and
     `getEndingSoonWatched`.
  2. Renders an **"Ending soon"** group at top (each row links to `/auction/[id]` with the
     time-left), then a **"Recent"** feed of stored events (icon + label by type, vehicle
     label, time-ago, link to `/auction/[id]` for `outbid`/`sold` or `/won/[id]` for `won`).
     Rows that were unread this visit get a highlight (e.g. a left-border/bg accent).
  3. Calls `markNotificationsRead(dealerId)` during render — the same "mutation during a GET
     render" pattern `/won/[id]` already uses with `close_auction`. Because the unread set was
     captured in step 1, the highlight reflects this visit and the clear takes effect next
     visit.
  - Empty state when there are no stored rows and nothing ending soon.
- Labels: `outbid` → "You were outbid on {vehicle}"; `won` → "You won {vehicle}"; `sold` →
  "Your {vehicle} sold". Amounts (current bid / sale price) come from the joined auction.

## 9. Security / grants

- `notifications`: `grant select on notifications to anon, authenticated;` (reads are
  dealer-scoped in the query — the same no-RLS demo trust model as `auctions`/`bids`/
  `dealers`); `grant select, insert, update, delete on notifications to service_role;`. **No**
  insert/update/delete to anon/authenticated — the only write paths are the SECURITY DEFINER
  RPCs (which run as owner) and `service_role` (tests).
- `mark_notifications_read`: SECURITY DEFINER, `set search_path = public`, service_role-only
  via revoke-before-grant; seller/recipient identity is always the httpOnly `dealer_id` cookie,
  never client input. Anon-denied regression test (mirrors Slices 2–6).
- `_notify` has its default PUBLIC `EXECUTE` revoked (`from public, anon, authenticated`) and
  is only reachable from within the definer writers (which call it as owner). Anon-denied is
  also enforced by the table: anon has no `INSERT` on `notifications`.

## 10. Error handling

- Generation is best-effort and transactional with the triggering write: the `_notify` insert
  runs inside the same RPC transaction as the bid/close/buy, so a notification is created iff
  the underlying event committed. No partial state.
- No dealer cookie on `/notifications` → redirect `/login` before any DB read.
- The badge and page are server-rendered from a snapshot; a bid landing between render and the
  next navigation simply shows on the next load. No optimistic client state to reconcile.

## 11. Testing

- **Integration (vitest vs local Supabase):** own-fixture hygiene — `createLiveAuction` +
  `deleteAuctions` (the shared helper is extended to also delete `notifications` referencing
  the auctions, before the auctions, since the FK does not cascade). `test_reset` (redefined
  here) truncates `notifications` between cases.
  - `place_bid` inserts `outbid` for the displaced leader when the lead changes; **no** row on
    a self-raise or a proxy-hold, and none when there was no prior leader.
  - `close_auction` inserts `won` (winner) + `sold` (seller) on a reserve-met close; **none**
    on a passed close.
  - `buy_now_listing` inserts `sold` for the seller and **no** `won` for the buyer.
  - `mark_notifications_read` sets `read_at` for the dealer's unread rows and leaves others
    untouched; anon-role call is denied (regression).
  - Lib: `getUnreadCount` (unread stored + ending-soon), `listNotifications` (join + order),
    `getEndingSoonWatched` (watched + in-window appears; out-of-window and unwatched do not —
    driven by `test_set_end_in_seconds`).
- **e2e (Playwright):** a two-dealer **outbid** flow on a **spare** live auction not asserted
  by any other spec (confirmed during planning; candidates in the a05/a07 range — `a01`
  bidding, `a02` discovery/realtime, `a03` buy-now, `a06` unpublish are all taken). Dealer A
  places a bid → dealer B outbids (re-login switches the cookie dealer) → dealer A opens
  `/notifications`, sees "You were outbid" and the badge, and the badge clears on a second
  navigation. `listing.spec` stays skipped (R2). `workers: 1`, fresh dev server per full run.
- **Invariant:** the anon client cannot execute `mark_notifications_read` or insert into
  `notifications`; a notification exists iff its triggering event (bid lead-change / sold
  close / buy-now) committed.

## 12. Workflow note

Implementation continues to target Claude Code web on a phone: small, well-bounded files,
managed services over self-hosted infra, repo as the single source of truth. Commits are a
single line with no co-author trailer; work on a feature branch and integrate via merge at
completion (never commit WIP directly to `main`). Credential-dependent tests stay deferred
until the app is code-complete — this slice adds none. The full vitest suite runs sequentially
(`fileParallelism: false`) because integration tests share one local Supabase DB; new
integration tests must reset/clean their own state and scope any global assertions to seeded
ids. e2e specs run with `workers: 1` and a *fresh* dev server per full run.
